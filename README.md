# GameJam.RedBarrel

# Технічна документація проекту: «Морський бій» (Unity 2D)

## 1. Загальний опис проекту
Реалізація класичної настільної гри «Морський бій» на рушії Unity з використанням мови C#. Проект побудований за архітектурним патерном **MVC (Model-View-Controller)**.

## 2. Архітектура та структура даних

### 2.1. Переліки (Enums)
Базові типи даних для опису станів гри:
* `ShipType`: Визначає типи кораблів та їхню довжину (`Battleship` = 4, `Cruiser` = 3, `Destroyer` = 2, `Boat` = 1).
* `CellState`: Визначає стан конкретної клітинки на полі (`Empty`, `Occupied`, `Miss`, `Hit`).
* `ShotResult`: Визначає наслідок пострілу (`Miss`, `Hit`, `Destroyed`).
* `GameState`: Стани машини станів гри (`PlayerTurn`, `EnemyTurn`, `GameOver`).

### 2.2. Математична логіка (Models)
Класи, що відповідають за чисту логіку:
* `Coordinate (struct)`: Інкапсулює координати X та Y. Містить логіку парсингу текстових координат.
* `Ship`: Зберігає інформацію про конкретний корабель, його тип, масив займаних координат та поточний рівень здоров'я (Health). Відповідає за отримання шкоди.
* `Board`: Центральний логічний клас ігрового поля 10х10. 
    * Містить двовимірний масив `CellState[,]` та список живих кораблів.
    * `CanPlaceShip()`: Алгоритм валідації розміщення, який сканує матрицю 3х3 навколо кожної палуби корабля для запобігання перетинам та дотикам.
    * `PlaceAllShipsRandomly()`: Алгоритм автоматичної генерації валідного флоту (10 кораблів) з використанням випадкових координат та орієнтацій (горизонтальна/вертикальна).
    * `ReceiveShot()`: Обробка пострілу, зміна стану клітинки та виклик зменшення здоров'я відповідного корабля.

### 2.3. Візуалізація (View)
Керування UI елементами Unity:
* `CellUI`: Скрипт-компонент префабу клітинки. Обробляє події натискання (`Button.onClick`) та змінює колір клітинки (білий, зелений, сірий, червоний) залежно від отриманого `CellState`.
* `BoardUI`: Генератор сітки. Інстанціює 100 префабів клітинок у `Grid Layout Group`, налаштовує їх координати та ідентифікує приналежність (поле гравця чи поле ворога).
* `UIManager`: Singleton-менеджер для виведення динамічних текстових повідомлень у центрі екрана за допомогою корутин (TextMeshPro).

### 2.4. Контролери (Controllers)
Зв'язують логіку з інтерфейсом:
* `GameManager`: Головний Singleton-контролер.
    * Ініціалізує логічні поля (`Board`) для гравця та ворога.
    * Керує машиною станів (передача ходу).
    * Реалізує логіку ШІ (штучного інтелекту) комп'ютера через корутину `EnemyTurnRoutine()`. Бот запам'ятовує обстріляні клітинки та отримує додаткові ходи при влучанні.
* `MenuManager`: Обробка UI головного меню (перехід на ігрову сцену, вихід з програми).
* `GameOverController`: Обробка екрана завершення гри. Використовує статичні дані з `GameManager` для визначення переможця та надає функціонал перезапуску циклу.

## 4. Організація сцен (Build Settings)
1.  **`MainMenu`** (Index 0): Стартова сцена з кнопкою запуску.
2.  **`GameplayScene`** (Index 1): Основна сцена гри. Містить два `BoardUI` контейнери, `EventSystem`, `UIManager` та `GameManager`.
3.  **`GameOverScene`** (Index 2): Фінальний екран з результатами (перемога/поразка) та маршрутизацією для нової гри.
